# Prompt: Build a Claude Code Monitor Dashboard

Copy everything inside the code fence below into Claude Code (or any capable
coding agent) to generate a project equivalent to this one. It is intentionally
detailed so the result is faithful — including the correct hook→status mapping,
session lifecycle, conversation-title extraction, the `.env` config, the
in-dashboard settings, and the `setup.sh` installer.

````text
Build a local-network "traffic light" monitor dashboard for Claude Code sessions.

## Overview
A Node.js server runs on my Mac. Claude Code hooks POST status updates to it, and
it pushes live state over WebSocket to a dark, landscape-optimized dashboard I
keep open on my phone (same Wi-Fi). Each session shows as a card with a colored
dot: green = idle/done, yellow = working, red = needs my input.

## Hard constraints
- Server dependencies: ONLY `express` and `ws`. No other npm packages
  (no dotenv, no nodemon, etc.). Parse `.env` yourself with a tiny loader.
- Dashboard: a single `public/index.html`, vanilla JS, no external libraries.
- Hook scripts: POSIX bash + `curl` + `python3` (preinstalled on macOS) only.

## Project structure
- server.js
- public/index.html
- hooks/_post.sh                (shared helper)
- hooks/on-session-start.sh
- hooks/on-user-prompt.sh
- hooks/on-pre-tool.sh
- hooks/on-post-tool.sh
- hooks/on-notification.sh
- hooks/on-stop.sh
- hooks/on-session-end.sh
- .claude/settings.json         (generated by setup.sh — git-ignored)
- .env                          (PORT config)
- setup.sh                      (installer)
- package.json
- README.md

## Configuration (.env)
A plain `.env` at the project root holds `PORT=2202`. The server reads it at
startup with a small built-in parser (an existing PORT env var overrides the
file). The hook scripts read the SAME `.env` so everything stays on one port.
The dashboard connects to whatever host/port served it, so it needs no port
config.

## Server (server.js)
Use express + ws only. Listen on `0.0.0.0` at PORT (from .env, default 2202).
HTTP server and WebSocket server share the same port.

Endpoints:
- POST /status — body { session_id, project, conversation, status }. Upsert into
  a Map keyed by session_id: { session_id, project, conversation, status,
  lastSeen }. Validate status ∈ {free, working, waiting}. Broadcast after change.
- DELETE /status/:id — remove a session, broadcast. (Used by SessionEnd and the
  dashboard's tap-to-dismiss.)
- GET / — serve public/index.html.
- WebSocket on the same port — on connect, immediately send the current sessions
  array; on every state change, broadcast the full sessions array to all clients.
  Message shape: { type: "sessions", sessions: [...] }.

IMPORTANT — session lifecycle:
- Do NOT prune sessions on a timer. A session must live until it is explicitly
  removed (DELETE via SessionEnd, or manual dismiss). Idle sessions stay visible.
- On startup, print the LAN IP and the phone URL (http://<ip>:<PORT>).

## Status mapping (this is the important part — get it right)
| Claude Code hook event            | Reported status | Color  |
|-----------------------------------|-----------------|--------|
| SessionStart                      | free            | green  |
| UserPromptSubmit                  | working         | yellow |
| PreToolUse                        | working         | yellow |
| PostToolUse                       | working         | yellow |
| Notification (permission_prompt)  | waiting         | red    |
| Notification (elicitation_dialog) | waiting         | red    |
| PreToolUse of AskUserQuestion / ExitPlanMode | waiting | red |
| Stop                              | free            | green  |
| SessionEnd                        | (delete session)| —      |

Rationale: red means "Claude is blocked asking ME to choose/confirm". That covers
two cases: (a) a permission/elicitation Notification, and (b) a blocking question
tool. `AskUserQuestion` and `ExitPlanMode` are TOOLS, so they arrive as a
PreToolUse (which would normally be "working") — the pre-tool script must inspect
`tool_name` and, for these blocking tools, report `waiting` instead of `working`.
After the user answers, PostToolUse flips it back to `working`. `Stop` means
Claude finished its turn and is idle = green. SessionEnd removes the card.

Note on parsing hook stdin: when extracting multiple fields in one pass, join them
with a NON-whitespace separator (e.g. ASCII Unit Separator \x1f), because shell
`read` collapses adjacent whitespace separators (tabs), which silently shifts
columns when a middle field (like transcript_path) is empty.

## Hook scripts
Claude Code passes hook context as JSON on stdin, including `session_id`, `cwd`,
and `transcript_path`. Implement a shared `hooks/_post.sh` that:
1. Resolves PORT: existing $PORT env var, else read PORT from ../.env, else 2202.
2. Reads stdin JSON and extracts session_id, cwd, transcript_path (use python3
   for robust JSON parsing; fall back sensibly if a field is missing).
3. Takes the desired status as $1: free|working|waiting|delete.
   - delete: curl -X DELETE http://localhost:$PORT/status/<session_id> and exit.
   - otherwise: derive project = basename(cwd); derive the conversation NAME from
     the transcript (see below); then curl -X POST .../status with
     { session_id, project, conversation, status }.
4. Fire-and-forget with a short timeout; never block or error out Claude Code.

Conversation name from the transcript: the transcript file (`transcript_path`)
is JSONL. Read ONLY the last ~128KB (for speed regardless of transcript size) and
pick, in order of preference: the latest line of type "ai-title" (field
`aiTitle`, the AI-generated conversation title), else the latest "last-prompt"
(field `lastPrompt`), else the first user message text. Collapse whitespace and
truncate to ~60 chars. Empty if nothing is available yet.

Thin wrappers (each just calls `_post.sh` with the right status):
on-session-start.sh→free, on-user-prompt.sh→working, on-pre-tool.sh→working,
on-post-tool.sh→working, on-notification.sh→waiting, on-stop.sh→free,
on-session-end.sh→delete. Make all scripts executable (chmod +x).

## Claude Code hook config (.claude/settings.json)
Wire all events with the correct Claude Code hooks format and ABSOLUTE paths:
SessionStart, UserPromptSubmit, PreToolUse (matcher "*"), PostToolUse
(matcher "*"), Notification (matcher "permission_prompt"), Stop, SessionEnd.
Because absolute paths are machine-specific, this file is generated by setup.sh
and git-ignored.

## Dashboard (public/index.html)
Single HTML file, vanilla JS, dark theme, optimized for landscape phone.

Cards: each session is a card showing a traffic-light dot, the conversation name,
the project name, the status label, a short session id, and "time ago". Cards lay
out in a wrapping row. If there are no sessions, show a centered "No active
sessions" message.

Traffic light colors: free #22c55e (green), working #eab308 (yellow, subtle pulse
animation), waiting #ef4444 (red, subtle pulse animation).

Features:
- WebSocket to ws://<same host>, auto-reconnect every 3s if disconnected.
- Connection indicator: a small dot top-left (green = connected, gray =
  reconnecting).
- Wake Lock: request navigator.wakeLock.request('screen') on load and re-acquire
  on visibilitychange.
- Fullscreen button (top-right): toggle document.documentElement.requestFullscreen.
  Handle iOS Safari, which has NO Fullscreen API: detect it and instead show a
  hint to use "Add to Home Screen". Add the iOS PWA meta tags
  (apple-mobile-web-app-capable etc.) so launching from the Home Screen runs
  truly fullscreen. Hide the button when already running standalone.
- Settings button (⚙, top-right) opening a small popover with controls, persisted
  to localStorage:
    * Card size: a range slider 60%–200% driving a CSS `--scale` variable that
      multiplies all card dimensions/fonts (use calc(... * var(--scale))).
    * Full width: a checkbox that makes each card span the full row.
    * Reset to defaults.
  Apply saved settings on load; tap outside to close the popover.
- Tap a card to dismiss it (confirm, then DELETE /status/:id) — useful because
  sessions are never auto-pruned.
- Everything updates via WebSocket; no page reloads.

## setup.sh (installer)
A bash script that makes onboarding portable, since hooks need absolute paths:
- Resolve the project root from the script's own location.
- chmod +x the hook scripts.
- Create a default .env (PORT=2202) if missing.
- Generate .claude/settings.json with absolute paths pointing at THIS clone.
- Offer to ALSO merge the hooks into the global ~/.claude/settings.json (so they
  fire for every project). The merge must: preserve the user's other settings and
  hooks, remove any prior install of THESE hook scripts (idempotent, even if the
  clone moved), back up the existing global file first, and never duplicate.
  Support flags: --global (no prompt) and --no-global.
- Use python3 for the JSON generation/merge.
````
